# tool-call-contract v0.2.0 Technical Design

## Overview

`tool-call-contract` v0.2.0 extends the v0.1 validation workflow into a captured-call regression harness.

The v0.1 architecture already separates contract definitions, config loading, validation, CLI orchestration, reporting, and artifact writes. v0.2.0 should build on that shape instead of replacing it:

- add optional capture suite and redaction config fields,
- resolve suite globs into deterministic file lists,
- enrich validation reports with grouping metadata,
- add a side-effect-free redaction planner with explicit write/check modes,
- and generate plain Vitest-compatible tests that use public package APIs.

The design keeps the library network-free and model-free. Captures remain untrusted JSON input. Config files remain trusted project code because loading TypeScript config already executes local code through `jiti`.

## Design Decisions

The v0.2.0 PRD left several choices open. This design resolves them as follows.

| Question                       | Decision                                                                                                                                                                        |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Capture suite matching         | Use a small glob dependency instead of an internal glob implementation. Prefer `tinyglobby` unless implementation discovers a better-maintained equivalent.                     |
| Generated test capture loading | Read JSON at runtime through `node:fs/promises` instead of importing JSON modules. This avoids JSON import assertion and test runner loader differences.                        |
| `redact` write behavior        | Mutate input files by default when no output option is provided. Support `--out`, `--out-dir`, `--dry-run`, and `--check` for safer flows.                                      |
| Multiple suites                | Allow repeated `--suite <name>` on `validate`, `redact`, and `generate-tests`.                                                                                                  |
| Report schema version          | Keep `schemaVersion: 1` and add optional report fields. Existing consumers can ignore the new metadata.                                                                         |
| Redaction target shape         | Redact raw parsed JSON before normalization. Dot paths are applied recursively to objects and arrays so users can target common capture shapes without provider-specific logic. |

## Runtime And Dependencies

Runtime targets remain unchanged:

- Node.js 20 or newer.
- TypeScript 5.5 or newer.
- Zod 4 as a peer dependency.

New runtime dependency:

- A tiny glob matcher for capture suite expansion.

The glob dependency should:

- support POSIX-style glob patterns on all platforms,
- return deterministic file ordering after sorting,
- support `cwd`,
- return files only,
- and avoid pulling in a large CLI framework or watcher stack.

The project should continue to avoid a CLI framework until argument parsing becomes materially more complex.

## Public API Changes

### Config Types

Extend `ToolCallContractConfig`:

```ts
export interface ToolCallContractConfig {
  contracts: readonly ToolContract[];
  outDir?: string;
  examples?: Record<string, readonly unknown[]>;
  include?: readonly string[];
  exclude?: readonly string[];
  captures?: CaptureSuiteConfig;
  redaction?: RedactionConfig;
}

export type CaptureSuiteConfig = Record<string, readonly string[]>;

export interface RedactionConfig {
  paths: readonly string[];
  replacement?: string;
}
```

`defineConfig` remains a light identity helper. Deep validation continues to live in `loadConfig`, matching v0.1.

### Loaded Config

`LoadedConfig` continues to expose the normalized config and resolved `cwd`/`outDir`.

`loadConfig` should normalize:

- `captures` to a record of non-empty suite names and string pattern arrays,
- `redaction.paths` to a string array,
- `redaction.replacement` to a string when provided,
- and default `redaction.replacement` to `"[REDACTED]"` only inside redaction execution helpers, not in user config.

Invalid config should continue to throw `ConfigLoadError` with `code: "config.invalid"` and precise messages.

### Validation Helpers

The existing validation helpers remain the public execution surface:

```ts
export function validateToolCall(...): ToolCallValidationResult;
export function validateToolCalls(...): ToolCallValidationResult[];
```

No generated test should import CLI internals. Tests generated by `generate-tests` should import `validateToolCalls` from `tool-call-contract` and the user's config file from a relative path.

### New Internal APIs

The implementation should add focused modules rather than growing `src/cli/app.ts`.

Recommended modules:

```text
src/captures.ts
src/redaction.ts
src/test-generation.ts
src/cli/redact.ts
src/cli/generate-tests.ts
```

`src/captures.ts` owns suite expansion and file deduplication.

```ts
export interface CaptureFileRef {
  path: string;
  suiteNames: readonly string[];
}

export interface ResolveCaptureFilesOptions {
  cwd: string;
  suites: readonly string[];
  files: readonly string[];
  captures?: CaptureSuiteConfig;
}

export interface ResolveCaptureFilesResult {
  files: CaptureFileRef[];
  findings: Finding[];
}
```

`src/redaction.ts` owns JSON path parsing, redaction planning, and deterministic formatting.

```ts
export interface RedactionPlanEntry {
  file: string;
  changed: boolean;
  content?: string;
  replacements: number;
  issues: ToolCallIssue[];
}

export interface RedactionPlan {
  entries: RedactionPlanEntry[];
  findings: Finding[];
}
```

`src/test-generation.ts` owns generated test rendering and write planning.

```ts
export interface GeneratedTestPlan {
  outFile: string;
  content: string;
  captureFiles: readonly CaptureFileRef[];
  findings: Finding[];
}
```

Only the public types that are useful to library consumers should be exported from `src/index.ts`. v0.2.0 can keep capture resolution, redaction planning, and test generation internal until the API proves stable.

## Capture Suite Resolution

### Config

Capture suites are optional:

```ts
export default defineConfig({
  contracts: [createIssue, searchDocs],
  captures: {
    smoke: ["captures/smoke/*.json"],
    regression: ["captures/regression/**/*.json"],
  },
});
```

Rules:

- Suite names must be non-empty strings after trimming.
- Suite patterns must be strings.
- Empty pattern arrays are valid config but produce `capture.suite-empty` when selected.
- Patterns resolve relative to loaded config `cwd`.
- Resolved paths are reported relative to `cwd` with forward slashes.
- Files are sorted lexicographically by relative path.
- Duplicate files are validated once per command.
- If a file belongs to multiple selected suites, `suiteNames` records every selected suite in CLI order, deduped.

### Direct Files And Suites

`validate`, `redact`, and `generate-tests` use a shared resolver:

- Direct file args are allowed.
- One or more `--suite <name>` flags are allowed.
- Direct files and suites may be combined.
- Unknown suite names produce an error finding with ID `capture.suite-unknown`.
- A selected suite with no matching files produces an error finding with ID `capture.suite-empty`.
- Direct files are included even when they do not belong to a suite.
- Direct files get `suiteNames: []`.

The `validate` command no longer requires a positional file if at least one suite was supplied.

## CLI Changes

### Command Names

Extend:

```ts
export type CommandName = "check" | "generate" | "validate" | "redact" | "generate-tests";
```

### Shared Options

Existing options stay intact:

```text
--cwd <path>
--config <path>
--json
```

Add common suite option:

```text
--suite <name>       Include a configured capture suite
```

`--suite` may be repeated.

### `validate`

Supported forms:

```sh
tool-call-contract validate captures/manual.json
tool-call-contract validate --suite smoke
tool-call-contract validate --suite smoke --suite regression captures/manual.json
```

Options:

```text
--allow-unknown      Downgrade unknown tool calls to warnings
--json               Print machine-readable output
```

Flow:

1. Parse CLI args.
2. Load config and registry.
3. Resolve suites and direct files through `resolveCaptureFiles`.
4. Validate each resolved file with the existing JSON parser and `validateToolCalls`.
5. Attach `file` to every result as today.
6. Build grouped validation metadata.
7. Apply finding policy.
8. Render report and resolve exit code with existing rules.

### `redact`

Supported forms:

```sh
tool-call-contract redact captures/raw.json
tool-call-contract redact --suite regression
tool-call-contract redact --suite regression --out-dir captures/redacted
tool-call-contract redact captures/raw.json --out captures/safe/raw.json
tool-call-contract redact --check --suite regression
tool-call-contract redact --dry-run --suite regression
```

Options:

```text
--suite <name>       Include a configured capture suite, repeatable
--out <path>         Write a single input file to a specific output file
--out-dir <path>     Write redacted files under a directory
--check              Fail if any file would change; do not write
--dry-run            Report planned changes; do not write
--json               Print machine-readable output
```

Rules:

- `redact` requires at least one direct file or suite.
- `redaction.paths` is required unless the user supplies a future CLI-only path option. v0.2.0 should not add CLI-only redaction paths.
- `--out` is valid only with exactly one resolved input file.
- `--out` and `--out-dir` are mutually exclusive.
- `--check` and write options are mutually exclusive.
- `--dry-run` may be combined with `--out` or `--out-dir` to preview destinations.
- With no output option, redaction writes changed content back to the original file.
- Redaction reads and writes JSON only.
- Malformed JSON becomes a result/finding and does not write.

Exit behavior:

- Exit `0` when no errors exist and, for `--check`, no file would change.
- Exit `1` when redaction would change files in `--check` mode or when redaction found invalid files/paths.
- Exit `2` for config/usage failures, matching current CLI behavior.

### `generate-tests`

Supported forms:

```sh
tool-call-contract generate-tests
tool-call-contract generate-tests --suite regression
tool-call-contract generate-tests --out test/tool-call-contract.generated.test.ts
```

Options:

```text
--suite <name>       Include a configured capture suite, repeatable
--out <path>         Output test file
--dry-run            Report planned write without writing
--json               Print machine-readable output
```

Rules:

- If no suite is supplied, include all configured capture suites.
- If no `captures` config exists, return `capture.suite-empty` with a clear message.
- Default output path is `test/tool-call-contract.generated.test.ts`.
- Output path resolves relative to `cwd`.
- Output paths must stay inside `cwd`.
- Generated test content is deterministic.
- Generated tests use runtime `node:fs/promises` JSON reads.
- Generated tests import the config through a relative path from the generated test file.
- Generated tests import package APIs from `"tool-call-contract"`.
- Generated test file URLs are computed relative to the generated test file, so custom `--out` locations work.

Generated test shape:

```ts
import { readFile } from "node:fs/promises";
import { describe, expect, it } from "vitest";

import { validateToolCalls } from "tool-call-contract";

import config from "../tool-call-contract.config";

const captureFiles = [
  {
    label: "captures/regression/create-issue.json",
    url: new URL("../captures/regression/create-issue.json", import.meta.url),
  },
] as const;

describe("tool-call-contract regression captures", () => {
  for (const file of captureFiles) {
    it(`validates ${file.label}`, async () => {
      const capture = JSON.parse(await readFile(file.url, "utf8"));
      const results = validateToolCalls(config.contracts, capture);
      expect(results).toEqual(expect.arrayContaining([expect.objectContaining({ ok: true })]));
      expect(results.every((result) => result.ok)).toBe(true);
    });
  }
});
```

The implementation can simplify the assertion to only `expect(results.every(...)).toBe(true)` if the more detailed assertion proves noisy. The generated file should prioritize readability over clever diagnostics; the CLI validation report remains the richer diagnosis path.

## Reporting

### Command Report

Extend `CommandReport` with optional fields:

```ts
export interface CommandReport {
  schemaVersion: 1;
  command: CommandName;
  success: boolean;
  summary: ReportSummary;
  findings?: Finding[];
  results?: ToolCallValidationResult[];
  artifacts?: ArtifactSummary;
  validation?: ValidationReportMetadata;
  redaction?: RedactionReportMetadata;
  generatedTests?: GeneratedTestReportMetadata;
}
```

Existing consumers can keep reading `findings`, `results`, `artifacts`, and `summary`.

### Validation Metadata

```ts
export interface ValidationReportMetadata {
  suites: Array<{
    name: string;
    files: string[];
    validResults: number;
    invalidResults: number;
  }>;
  files: Array<{
    path: string;
    suiteNames: string[];
    validResults: number;
    invalidResults: number;
  }>;
  contracts: Array<{
    name: string;
    validResults: number;
    invalidResults: number;
    unknownResults: number;
  }>;
}
```

The grouping builder should accept resolved files plus validation results and produce stable sorted output.

Sorting rules:

- Suites retain selected CLI order.
- Files sort by path.
- Contracts sort by contract name, with unknown calls grouped under the observed call name when available and `"unknown"` last.

### Redaction Metadata

```ts
export interface RedactionReportMetadata {
  checked: boolean;
  dryRun: boolean;
  files: Array<{
    path: string;
    destination?: string;
    changed: boolean;
    replacements: number;
  }>;
}
```

`redact --check` should add a `redaction.would-change` error finding for each file that would change.

### Generated Test Metadata

```ts
export interface GeneratedTestReportMetadata {
  outFile: string;
  dryRun: boolean;
  captureFiles: string[];
  created: boolean;
  updated: boolean;
  unchanged: boolean;
}
```

This can either be a dedicated field or reuse `artifacts`. Prefer a dedicated field because generated tests are user-owned files, not manifest-owned generated artifacts.

### Human Output

Human validation output should add summary context before per-result lines:

```text
Validation suites:
  regression: 3 file(s), 8 valid, 1 invalid

Validation files:
  captures/regression/create-issue.json: regression, 4 valid, 0 invalid
```

Human redaction output should report:

```text
Redaction: 2 changed, 1 unchanged.
  changed captures/raw.json: 3 replacement(s)
```

Human generated-test output should report:

```text
Generated test: test/tool-call-contract.generated.test.ts updated.
  Captures: 3 file(s)
```

## Redaction Design

### Dot Path Syntax

Supported syntax:

```text
arguments.email
arguments.customer.ssn
metadata.authorization
tool_calls.function.arguments.apiKey
```

Rules:

- Segments are separated by `.`.
- Empty segments are invalid.
- `*` matches every element of an array or every property of an object.
- Numeric segments match array indexes or object keys.
- No escaping is supported in v0.2.0.
- JSONPath, recursive descent, and regex matching are out of scope.

Path examples:

```text
calls.*.arguments.email
choices.*.message.tool_calls.*.function.arguments
output.*.arguments.apiKey
```

### Recursive Application

Because capture wrappers differ by provider, redaction should apply each dot path from every object node in the JSON tree, not only the root.

For example, `arguments.email` should match:

```json
{ "name": "create_issue", "arguments": { "email": "a@example.com" } }
```

and also:

```json
{
  "calls": [{ "name": "create_issue", "arguments": { "email": "a@example.com" } }]
}
```

This keeps paths understandable while covering common nested capture arrays.

### Replacement

Replacement is always a JSON string in v0.2.0.

Default:

```json
"[REDACTED]"
```

When a path matches, replace the full value at that path with the replacement string. Do not partially redact strings and do not inspect values.

### Formatting

Redacted JSON should use the existing `formatJson` helper from `src/artifacts.ts` so formatting remains deterministic:

- two-space indentation,
- trailing newline,
- stable `JSON.stringify` behavior.

Key ordering should preserve JavaScript object insertion order from the parsed JSON. v0.2.0 should not sort user capture keys because that creates larger diffs than necessary.

## File Safety

Path safety rules:

- All suite-resolved files must live under `cwd`.
- Direct files resolve relative to `cwd`; paths outside `cwd` should produce an error finding.
- `--out-dir` resolves relative to `cwd` and must stay under `cwd`.
- `--out` resolves relative to `cwd` and must stay under `cwd`.
- Generated test output must stay under `cwd`.
- Redaction should write by first producing complete content in memory and then writing the file with `writeFile`.

The existing artifact manifest deletion safety model does not apply to redaction or generated tests because these are user-owned files.

## Finding IDs

Use the following finding IDs:

```text
capture.suite-unknown
capture.suite-empty
capture.file-outside-root
capture.file-read-failed
capture.file-invalid-json
redaction.config-missing
redaction.path-invalid
redaction.would-change
redaction.write-failed
generated-test.no-captures
generated-test.outside-root
generated-test.write-failed
```

Keep existing validation issue codes for per-call results:

```text
file.read-failed
file.invalid-json
call.unsupported-shape
call.unknown-tool
call.invalid-json-arguments
```

The implementation may keep existing `file.read-failed` and `file.invalid-json` result issue codes for validation compatibility, while using `capture.file-*` finding IDs only for suite/redaction planning failures.

## Testing Strategy

### Unit Tests

Add focused tests for:

- config normalization of `captures`,
- config normalization of `redaction`,
- suite glob resolution,
- deterministic file ordering and deduplication,
- unknown and empty suite findings,
- dot-path parsing,
- recursive redaction behavior,
- `*` wildcard behavior,
- redaction check mode,
- generated test rendering,
- output path safety.

### CLI Tests

Extend CLI tests for:

- `validate --suite regression`,
- `validate --suite one --suite two`,
- `validate --suite regression captures/manual.json`,
- `redact --check --suite regression`,
- `redact --out-dir captures/redacted --suite regression`,
- `generate-tests --suite regression --out test/tool-call-contract.generated.test.ts`,
- usage failures for invalid option combinations.

### E2E Tests

Update the basic example project to include:

- a `smoke` suite,
- a `regression` suite,
- at least one capture requiring redaction,
- and a generated-test workflow.

The e2e test should:

1. Run `check`.
2. Run `generate`.
3. Run `validate --suite smoke`.
4. Run `redact --check --suite regression` against an already redacted fixture.
5. Run `generate-tests --suite regression`.
6. Verify the generated test file exists and contains the expected capture file names.

Running the generated Vitest test inside the fixture project is useful but not required for v0.2.0 if it adds brittle test-runner bootstrapping. The primary package test suite should still validate generated code shape.

## Documentation Updates

Update:

- README capture validation section,
- README CLI reference,
- example config,
- example capture files,
- `docs/v0.2.0/implementation-plan.md` after this design is accepted.

Docs must state:

- Redaction is deterministic replacement, not PII detection.
- Users must choose paths appropriate to their capture format.
- Generated tests do not call models or execute tool handlers.
- `validate --suite` is the recommended CI entry point for captured regressions.

## Migration And Compatibility

Existing v0.1 configs remain valid because all new fields are optional.

Existing `validate <files...>` behavior remains valid:

- direct file validation still works,
- result entries retain `file`,
- `--allow-unknown` still downgrades unknown tools,
- JSON report consumers can ignore new optional metadata.

The only parser behavior change is that `validate` accepts no positional files when one or more suites are provided.

## Release Notes Shape

The v0.2.0 release should be described as:

- capture suites for repeatable validation commands,
- deterministic redaction for commit-safe captures,
- generated Vitest regression tests,
- grouped validation reports,
- and expanded example/docs coverage.

Avoid positioning v0.2.0 as a provider adapter release. The release deepens the existing contract-testing loop rather than broadening provider support.
